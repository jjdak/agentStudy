# 03. 使用中如何降低幻觉影响

> 适用场景：软件部门内部 AI 培训
> 目标听众：普通软件工程师
> 建议时长：20～30 分钟，可作为现场练习材料
> 使用边界：当前只能在工作环境之外使用 AI 工具，因此不得输入公司源码、内部日志、芯片结构、接口协议、客户问题或任何可反推出产品信息的材料。

## 1. 一句话原则

使用 AI 时，不要问“它会不会幻觉”，而要问：

> 这个任务是否脱敏？假设是否明确？输出是否可验证？错误是否可控？

## 2. 外部 AI 工具的安全边界

在当前约束下，外部 AI 适合做通用辅助，不适合接触公司内部材料。

### 2.1 不应输入的内容

- 公司源码；
- 内部日志；
- 芯片结构、寄存器、微架构、接口协议；
- 内部 bug、波形、客户问题；
- 未公开 benchmark；
- 内部调度策略、验证策略、性能瓶颈；
- 任何可能反推出产品信息的材料。

### 2.2 可以使用的方向

- 通用脚本模板；
- 与芯片结构无关的结构体转换脚本；
- 通用调度器代码练习；
- 公开语言特性解释；
- 脱敏文档改写；
- 通用测试样例；
- 公开库/API 的用法查询；
- 学习材料、培训大纲、PPT 草稿。

一句适合培训中强调的话：

> 降低幻觉不能成为扩大输入范围的理由。安全边界优先于效率。

## 3. 任务风险分级

### 3.1 低风险任务

适合让 AI 生成初稿：

- 生成通用脚本；
- 解释公开语言特性；
- 改写脱敏文档；
- 生成 README 初稿；
- 生成通用单元测试样例；
- 写与公司信息无关的小工具。

验证方式：

- 人工阅读；
- 本地运行；
- lint / type check；
- 不依赖模型给出的事实性引用。

### 3.2 中风险任务

可以使用，但必须验证：

- 生成通用调度器代码；
- 多线程/异步逻辑；
- 使用第三方库 API；
- 性能优化建议；
- 边界条件处理；
- 复杂脚本或自动化流程。

验证方式：

- 查官方文档；
- 固定库版本；
- 编写单元测试；
- 构造反例；
- 要求模型列出假设；
- 要求模型标注未验证项。

### 3.3 高风险任务

不应交给外部 AI：

- 公司项目代码；
- 芯片相关设计信息；
- 内部问题定位；
- 真实客户场景；
- 内部性能数据；
- 可以影响真实产品决策的结论。

## 4. 提问时如何降低幻觉

### 4.1 明确任务边界

不推荐：

```text
帮我写一个调度器。
```

推荐：

```text
请用 Python 写一个教学用途的非抢占式任务调度器。
输入任务包含 id、priority、deadline、duration。
这是脱敏练习，不涉及真实项目。
请先列出你的假设，再给代码。
如果无法保证最优，请明确说明。
```

### 4.2 要求模型列出假设

可以直接加：

```text
回答前请先列出你做出的假设。
哪些是用户明确给出的？
哪些是你为了完成任务自行补充的？
```

这一步能显著降低“模型偷偷补条件”的风险。

### 4.3 要求区分事实、推断和建议

```text
请把回答分成三部分：
1. 已知事实；
2. 基于事实的推断；
3. 建议验证步骤。
不要把推断写成事实。
```

### 4.4 要求承认不确定

```text
如果信息不足，请直接说不确定。
不要为了完整性编造 API、版本号、论文、命令或参数。
```

### 4.5 要求引用来源

适用于公开 API、标准、库文档：

```text
请优先依据官方文档回答。
如果没有找到可靠来源，请说明没有验证。
```

注意：引用也可能被编造。看到引用后仍要打开链接或查官方文档核对。

## 5. 上下文组织模板

给模型材料时，可以使用下面结构：

```text
【任务目标】
生成一个教学用的小工具，用于演示任务调度。

【已知约束】
- 不涉及公司项目；
- 使用 Python 标准库；
- 不允许引入第三方依赖；
- 非抢占式调度；
- deadline 无法满足时需要报告。

【输入格式】
...

【输出格式】
...

【禁止事项】
- 不要编造第三方库；
- 不要声称已运行测试；
- 不要使用真实公司名称或内部术语。

【期望输出】
1. 假设；
2. 实现；
3. 单元测试；
4. 未验证风险。
```

这种格式的价值是把“模型需要猜”的空间显式压小。

## 6. 验证手段

### 6.1 代码类输出

不要只看解释，要运行：

- 编译；
- 单元测试；
- lint；
- type check；
- 静态分析；
- 边界条件测试；
- 失败路径测试。

### 6.2 文档类输出

检查：

- 是否混入内部信息；
- 是否把推测写成事实；
- 引用是否存在；
- 日期和版本是否准确；
- 是否存在过度承诺。

### 6.3 技术建议类输出

要求模型给出：

- 前提条件；
- 适用场景；
- 不适用场景；
- 验证方法；
- 替代方案；
- 风险和副作用。

## 7. Agent 工具的验收规范

Claude Code、Codex、OpenClaw 这类 agent 可以读文件、运行命令、调用工具，因此在可验证任务上通常比普通聊天更可靠。但它们也可能误用工具、误读结果或执行错误动作。

建议要求 agent 完成任务后必须说明：

```text
1. 修改了哪些文件；
2. 运行了哪些命令；
3. 哪些测试通过，哪些没跑；
4. 哪些结论来自工具输出；
5. 哪些结论只是模型推断；
6. 还有哪些风险或未验证项。
```

对下面动作保留人工确认：

- 删除文件；
- 覆盖用户修改；
- 安装依赖；
- 访问网络；
- 提交 commit；
- push 到远端；
- 调用外部服务；
- 修改配置、脚本、CI、部署相关文件。

一句话总结：

> Agent 会降低“可被工具反馈验证的问题”的幻觉，但不会消灭幻觉。它把可靠性从“模型一次回答是否正确”，变成“工具反馈、测试、权限、审计和人工 review 是否形成闭环”。

## 8. 现场练习：脱敏调度器

可以用完全虚构、脱敏的例子：

```text
请用 Python 生成一个任务调度器。
要求：
1. 任务有 id、priority、deadline、duration；
2. 不允许抢占；
3. 输出调度顺序；
4. 如果 deadline 无法满足，需要报告原因。
```

### 8.1 第一轮：直接生成

观察点：

- 是否说明调度策略假设；
- 是否处理相同 priority；
- 是否处理 deadline 冲突；
- 是否给出测试；
- 是否承认这是启发式而非最优。

### 8.2 第二轮：加入反幻觉提示

```text
请先列出你做出的调度假设。
如果问题在算法上无法保证最优，请明确说明。
代码生成后，请给出 5 个单元测试，包括 deadline 不可满足的反例。
不要使用不存在的第三方库。
```

观察点：

- 幻觉是否减少；
- 假设是否更清楚；
- 错误是否更容易被测试抓住。

### 8.3 第三轮：加入验证闭环

```text
请把代码和测试分开输出。
测试应覆盖：
1. 空任务列表；
2. priority 相同；
3. deadline 冲突；
4. duration 为 0；
5. 大量任务的稳定排序。
最后请指出这段实现不能保证哪些性质。
```

培训结论：

> 降低幻觉的关键不是相信更强模型，而是把任务设计成可验证。

## 9. 常用提示词模板

### 9.1 代码生成模板

```text
请生成一段教学用途的代码。
约束：
- 不使用公司信息；
- 不引入第三方依赖，除非我明确允许；
- 先列出假设；
- 再给实现；
- 再给测试；
- 最后列出未验证风险。
如果你不确定某个 API 是否存在，请标注为需要查证，不要编造。
```

### 9.2 代码 review 模板

```text
请从以下角度 review 这段脱敏代码：
1. 边界条件；
2. 错误处理；
3. 资源释放；
4. 并发风险；
5. 可测试性；
6. 是否依赖未说明的假设。
请区分“确定问题”和“建议检查”。
```

### 9.3 文档改写模板

```text
请改写下面这段脱敏文字，使其更适合内部培训。
要求：
- 不添加新的事实；
- 不引入公司、产品或芯片相关细节；
- 如果原文信息不足，请保留模糊表达；
- 不要编造数据或结论。
```

### 9.4 根因分析模板

```text
请基于以下脱敏现象列出可能原因。
要求：
- 不要断言唯一根因；
- 每个原因给出支持证据和反证；
- 给出下一步验证方法；
- 信息不足时明确说明。
```

## 10. 输出验收清单

使用 AI 输出前，可以按下面清单过一遍：

- [ ] 输入材料是否已经脱敏？
- [ ] 是否包含公司源码、日志、芯片结构或内部接口？
- [ ] 模型是否列出了假设？
- [ ] 模型是否区分事实、推断和建议？
- [ ] 是否存在编造 API、包名、版本号、论文或标准条款？
- [ ] 是否运行过必要测试？
- [ ] 测试是否覆盖失败路径和边界条件？
- [ ] 是否查看了官方文档？
- [ ] 是否有人工 review？
- [ ] 输出是否适合分享给同事？

## 11. PPT 可用总结

### AI 适合做什么

- 生成初稿；
- 提供思路；
- 写通用脚本；
- 解释公开技术；
- 生成测试样例；
- 做脱敏文档润色。

### AI 不适合直接做什么

- 替代事实核查；
- 替代代码 review；
- 替代安全审查；
- 处理公司敏感材料；
- 对高风险问题直接下结论。

### 降低幻觉的工作流

```text
脱敏输入 → 明确约束 → 列出假设 → 生成初稿 → 工具验证 → 人工 review → 小范围使用
```
