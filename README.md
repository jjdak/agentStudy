# Coding Agent 个人学习路线

这是一套面向软件工程师的个人学习材料，目标不是覆盖所有 Agent 理论，而是回答一个更实际的问题：

> 怎样把 Coding Agent 当作能力很强、但需要约束和验证的代码贡献者使用？

学习材料只使用公开、虚构或明确授权的内容。当前工作环境之外使用 AI 时，不应提交商业秘密、非公开代码、芯片结构、内部日志、账号凭证、个人信息或其他未经授权的数据。

## 学完应当具备的能力

完成主线后，你应该能够：

1. 解释 LLM 为什么会产生幻觉，以及工具反馈为什么只能降低部分错误；
2. 判断一个任务应该使用脚本、普通聊天、固定工作流还是 Agent；
3. 把模糊需求写成包含边界、权限、验收和停止条件的任务合同；
4. 面对未知 Bug 时，让 Agent 先建立证据，再进行最小修改；
5. 判断 Agent 修复的是根因 A，还是只在报错位置 B 掩盖了症状；
6. 使用 Git diff、编译器、测试、Sanitizer 和独立评分器检查结果；
7. 在 Linux 隔离环境中，用同一标准任务重复测试不同模型或 Agent。

## 建议学习安排

主线约 2～3 小时，可拆成六次：

| 次数 | 时间 | 内容 | 产出 |
|---:|---:|---|---|
| 1 | 25 分钟 | 语言模型、Transformer 与幻觉 | 用自己的话解释“流畅不等于真实” |
| 2 | 25 分钟 | Agent、工具反馈与适用边界 | 为 5 个任务选择合适自动化级别 |
| 3 | 30 分钟 | 任务合同与上下文组织 | 写出一个可执行任务要求 |
| 4 | 30 分钟 | 根因诊断与最小修复 | 写出 A 处产生、B 处表现的调试协议 |
| 5 | 30 分钟 | 权限、验证、成本与复盘 | 完成一页检查清单 |
| 6 | 30～60 分钟 | SWE-bench C++ 练习 | 得到 patch、评分报告和复盘记录 |

先读 [01：必要原理](docs/01_foundations.md)，再把 [02：实战手册](docs/02_coding_agent_playbook.md) 作为日常主线。真正执行任务时打开 [03：个人实践](docs/03_personal_practice.md)。

## 三条贯穿全仓库的原则

1. **Agent 只能提交候选结果，不能批准自己的结果。** “已完成”“测试通过”和解释都需要外部证据。
2. **先找第一次不变量被破坏的位置，再修改最终报错位置。** 现象消失不等于根因修复。
3. **先使用最低必要复杂度。** 能用确定性程序解决的任务，不升级为 Agent；单 Agent 足够时，不增加多 Agent。

## 维护边界

- 原理和方法以稳定资料为主，资料核对日期为 **2026-07-19**。
- 模型名称、价格、上下文长度、产品权限和榜单成绩变化较快，不进入主学习路线；使用前查阅官方最新资料。
- 标准练习固定上游 commit、数据集 revision 和容器 digest。单个公开任务只能作为学习和 smoke test，不能代表完整模型能力。
- `lab/**/.runtime/`、`lab/**/runs/` 和离线镜像包不提交 Git。
