# 03. 个人实践：模板、检查清单与标准任务

> 学习目标：把前两篇的方法转化为可复制指令和一次真实、可评分的练习。建议阅读与实践 45～75 分钟。

## 1. 一页执行清单

### 开始前

- [ ] 输入是公开、虚构或明确授权的材料；
- [ ] 任务适合 Agent，而不是十几行脚本即可完成；
- [ ] 目标、非目标和可观察交付物已经写清；
- [ ] 已定义原始复现、回归测试和完成标准；
- [ ] 已记录基线 commit、`git status` 和用户已有修改；
- [ ] Agent 只写独立工作区，默认无网络、无凭证、无 push 权限；
- [ ] 测试、评分器和参考答案不在 Agent 可写范围内；
- [ ] 已设置停止条件、时间预算和最大重试次数。

### 执行中

- [ ] 先建立基线，再进行修改；
- [ ] 未知 Bug 先只读诊断，区分事实、假设和未知；
- [ ] 寻找第一次不变量被破坏的位置，而不是只看最终报错；
- [ ] 每个候选根因都有支持证据、反证和可证伪实验；
- [ ] 修改保持小而集中，没有弱化测试、吞错或扩大权限；
- [ ] 工具失败时检查命令、环境和退出码，不把尝试当通过；
- [ ] 连续两轮没有新增证据时停止并复盘。

### 接受前

- [ ] 检查完整 diff，包括测试、CI、依赖、配置和锁文件；
- [ ] 原始复现和根因定向测试在旧版失败、新版通过；
- [ ] 构建、项目测试和适用的静态/动态检查通过；
- [ ] 独立评分从干净基线应用 patch，而不是复用 Agent 工作区；
- [ ] 日志、退出码、patch hash、环境和未验证项已经保存；
- [ ] 人工确认实现符合真实意图并明确接受剩余风险。

## 2. 可复制模板一：未知 Bug 的根因诊断

```text
目标：诊断并修复下面的可复现问题。当前不知道根因；报错位置不一定是错误产生位置。

【已知事实】
- 现象：<完整描述>；
- 复现命令和输入：<命令/输入>；
- 环境：<OS、编译器、标准、依赖版本>；
- 已有日志：<路径或内容>。

【范围与权限】
- 允许读取：<路径>；
- 允许修改：<路径>；
- 当前阶段只读，不修改源码；
- 不访问网络、不安装依赖、不修改测试/CI、不执行 Git 提交或 push。

【阶段 1：诊断】
1. 稳定复现并记录命令、退出码和完整错误；
2. 使用已有测试、编译器告警、Sanitizer、调试器或静态分析寻找第一次非法状态；
3. 区分错误产生位置 A 和检测位置 B；
4. 对每个候选根因列出支持证据、反证和一个可证伪实验；
5. 使用“第一次非法状态—违反的不变量—传播路径—B 为何只是检测位置”的格式汇报；
6. 证据不足时明确写未知和下一步，不为了给答案而修改；
7. 汇报后暂停，等待我确认是否进入修复阶段。

【阶段 2：最小修复】
- 只修复第一次破坏不变量的位置；
- 禁止删除检查、吞错、无条件返回成功、跳过释放、关闭 Sanitizer 或增加 suppression；
- 如需修改 B，必须证明 B 还有独立缺陷。

【完成标准】
- 原始复现通过；
- 根因定向测试在旧版失败、新版通过；
- 适用的构建、回归和 Sanitizer 检查通过；
- B 原有检查和清理仍然执行；
- 临时撤销关键修复后，根因测试重新失败；
- 输出 diff、实际命令、退出码、未运行项和剩余风险。
```

## 3. 可复制模板二：实现一个可验证功能

```text
目标：在 <仓库> 中实现 <可观察行为>。

背景：
- 当前行为：<事实>；
- 期望行为：<示例和反例>；
- 环境与版本：<明确版本>。

非目标：
- 不重构 <范围>；
- 不改变 <API/ABI/格式/兼容行为>；
- 不增加新依赖。

工作方式：
1. 只读检查相关代码、测试和构建说明；
2. 返回需求理解、关键歧义、修改计划和验证计划；
3. 没有阻塞歧义后进行最小修改；
4. 先运行针对性测试，再运行项目规定的完整检查；
5. 任一强制检查失败时不能宣称完成。

权限：
- 只修改：<allowlist>；
- 不联网、不安装、不提交、不 push；
- 测试、CI、依赖和配置变化需要单独确认。

验收：
- <功能用例>；
- <边界和错误路径>；
- <兼容性/性能/安全不变量>；
- <canonical 构建和测试命令>；
- 最终报告包含 diff、命令、退出码、未验证项和风险。

停止条件：
- 需求存在会改变方案的歧义；
- 需要越出文件范围或提升权限；
- 连续两轮没有新增证据；
- 环境无法执行规定检查。
```

## 4. 标准练习：SWE-bench Multilingual C++ 任务

练习直接使用 [SWE-bench Multilingual](https://www.swebench.com/multilingual.html) 的真实任务 `fmtlib__fmt-2310`。它不采用人为构造的五类 Bug，也不把 Prompt 长短作为实验变量：

- 上游项目：[`fmtlib/fmt`](https://github.com/fmtlib/fmt)；
- 原始问题：[`fmtlib/fmt#2310`](https://github.com/fmtlib/fmt/issues/2310)；
- 任务类型：数值格式化的真实缺陷修复；
- 基线：修复前的公开仓库 commit；
- 评分：SWE-bench 的问题修复测试与回归测试；
- 环境：官方 `linux/x86_64` Docker 镜像；
- 隔离：Agent 不读取测试补丁、参考补丁和评分日志。

任务选择依据、公开通过率和版本固定方式见 [基准选择记录](../references/benchmark_selection.md)。

### 4.1 为什么选它

它具备几个适合学习的条件：

1. 问题来自真实开源项目，不在源码中标注根因；
2. 需要理解格式化语义和已有实现，不能只修一条显眼断言；
3. 标准评分包含问题修复测试和 102 个回归测试；
4. 在截至 2026-07-19 的 13 组公开 SWE-bench 模型/Agent 结果中仅 2 组解决；
5. 官方任务镜像已发布，可按 digest 固定并离线迁移。

难度数据只能说明它在这些公开设置中较难，不能保证当前强模型一定失败。任务已经公开多年，也可能进入模型训练数据。因此它适合练习和 smoke test，不适合单独证明一个内部模型的整体编码能力。

### 4.2 练习流程

```text
联网 Linux 准备评分器和固定镜像
        ↓
从官方镜像导出修复前源码，移除远端和 Git 历史
        ↓
只把工作区和固定任务要求交给 Agent
        ↓
Agent 调查、修改并运行仓库公开检查
        ↓
收集纯 Git patch
        ↓
评分器从干净官方镜像应用 patch 和隐藏测试
        ↓
记录 resolved、日志、耗时、patch 和人工复盘
```

Agent 运行公开检查时，可由受限包装器进入固定工具链；不需要向 Agent 开放 Docker socket。最终评分仍由另一个进程从干净镜像执行。

具体命令见 [练习 README](../lab/swebench-fmt-2310/README.md)。

### 4.3 练习时不要偷看什么

被测 Agent 不应访问：

- SWE-bench 数据集中的 `patch` 和 `test_patch`；
- 上游修复 commit、PR、Git tag 或完整远端历史；
- 其他模型的 patch 和执行轨迹；
- `.runtime/` 中的本地数据集与评分器；
- `runs/<id>/evaluation/` 中生成的 `eval.sh` 和评分日志。

即使这些资料本身是公开的，也必须在本次运行中隔离，否则无法判断 Agent 是解决问题还是读取答案。

## 5. 一次练习如何复盘

不要只记录 PASS/FAIL。完成后填写：

```text
任务 ID：
模型与版本：
Agent/脚手架版本：
运行日期和环境：
权限与网络状态：
总耗时、模型调用和费用：

1. Agent 第一次提出的根因是什么？
2. 哪条证据支持或否定了它？
3. 它是否区分已运行和未运行的检查？
4. 修改是否集中在第一次破坏语义的位置？
5. 是否出现无关修改、弱化测试或伪验证？
6. 公开检查结果是什么？
7. 独立 SWE-bench 评分结果是什么？
8. 人工 review 最耗时的部分是什么？
9. 即使 resolved，仍有哪些风险？
10. 下一次任务合同需要调整什么？
```

### 过程质量只看可观察证据

不要要求模型暴露私有思维过程，也不要用“思考文字长”判断质量。评估以下可观察产物：

- 根因主张是否对应源码和工具证据；
- 是否设计了能够否定自己假设的实验；
- 实际命令、退出码和环境是否完整；
- 是否如实说明未验证项；
- patch 是否小而集中；
- 人能否快速复现和接管。

## 6. 在内网比较模型时

同一个任务可以作为接入和流程 smoke test，但不应把单题结果写成“模型 A 比模型 B 强”。如果要做内部比较，至少做到：

1. 固定任务文本、base commit、容器 digest、评分器和硬件资源；
2. 固定 Agent 脚手架、工具、权限、上下文预算和超时；
3. 每个模型独立使用干净工作区，不共享历史和缓存答案；
4. 同一设置至少重复 3 次，记录偶然成功和稳定性；
5. 评分只使用模型外的测试和工具，不用另一个 AI 判定正确；
6. 同时记录 resolved、patch 是否可应用、越界修改、耗时、费用/调用量和人工复核时间；
7. 后续扩展多个不同仓库和缺陷类型，再讨论能力覆盖面。

建议结果表：

| run | 模型 | Agent 版本 | resolved | patch 可应用 | 越界修改 | 墙钟时间 | 调用/费用 | 人工复核时间 |
|---|---|---|---:|---:|---:|---:|---:|---:|

公开 benchmark 存在污染、测试不完整和环境缺陷的可能。内部报告应同时写明：

- 这是公开任务，可能被训练或记忆；
- 通过只表示达到该任务的测试 oracle；
- 单个任务不能代表总体能力；
- Agent 脚手架和权限设置也是实验变量。

## 7. 个人能力自查

完成练习后，确认自己能回答：

1. 为什么这个任务适合 Agent，而不是普通聊天？
2. Agent 可见内容与评分器内容如何隔离？
3. 如果 Agent 只让公开测试变绿，为什么还不能接受？
4. 怎样证明根因修复，而不是症状隐藏？
5. 哪些主张由编译器、测试、人工 review 分别支持？
6. 怎样把同一环境离线迁移到内网 Linux？
7. 为什么一次 resolved 不能代表模型能力？

如果这些问题能够结合自己的运行日志回答，学习目标就已经从“会写 Prompt”转向“会设计、监督和验收 Agent 任务”。
